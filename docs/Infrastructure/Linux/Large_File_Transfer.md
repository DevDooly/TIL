# 대량 파일 전송 가이드 (rsync 활용)

NAS 간에 수백만 개의 파일을 전송해야 할 때, 특히 운영 중인 서버에서 시스템 부하(I/O)를 최소화하면서 안전하게 옮기는 방법을 다룹니다.

---

## 1. rsync를 선택하는 이유

`cp`나 `mv` 명령어는 대량 파일 작업 시 네트워크 단절이나 시스템 중단에 취약하며, 현재 진행 상태를 알기 어렵습니다. 반면 **rsync**는 다음과 같은 장점이 있습니다.

* **증분 전송**: 변경된 부분이나 누락된 파일만 골라서 전송 가능 (중단 후 재시작 시 매우 유리).
* **부하 조절**: 대역폭 제한(`--bwlimit`) 및 I/O 우선순위 조절 가능.
* **무결성 검사**: 체크섬을 통해 데이터가 정확히 복사되었는지 확인 가능.

---

## 2. 실전 전송 시나리오

**상황**: `/A/20260318` 폴더를 `/B` 아래로 복사하여 최종적으로 `/B/20260318` 구조를 만듦.

### 2.1 I/O 부하 최소화 핵심 명령어

```bash
ionice -c 3 rsync -avh --bwlimit=10000 --progress /A/20260318 /B/
```

### 2.2 주요 옵션 설명

* **`ionice -c 3`**: 이 프로세스의 I/O 우선순위를 'Idle'로 설정합니다. 다른 프로세스가 디스크를 쓰지 않을 때만 작업을 수행하므로 운영 중인 서비스에 미치는 영향을 최소화합니다.
* **`-a` (archive)**: 권한, 심볼릭 링크, 수정 시간 등 모든 속성을 유지하며 복사합니다.
* **`-v` (verbose)**: 진행 과정을 상세히 출력합니다.
* **`-h` (human-readable)**: 숫자를 읽기 쉬운 단위(GB, MB 등)로 표시합니다.
* **`--bwlimit=10000`**: 전송 대역폭을 초당 10,000KB(약 10MB/s)로 제한합니다. 네트워크 및 디스크 I/O가 폭주하는 것을 방지합니다.
* **`/A/20260318` vs `/A/20260318/`**:
    * 끝에 슬래시(`/`)가 **없으면**: `20260318` 폴더 **자체**를 복사합니다. (결과: `/B/20260318/...`)
    * 끝에 슬래시(`/`)가 **있으면**: 폴더 **내부의 내용물**만 복사합니다. (결과: `/B/...`)
    * *따라서 본 시나리오에서는 슬래시 없이 사용해야 합니다.*

---

## 3. 안정적인 작업을 위한 추천 스크립트

100만 개 이상의 파일을 옮길 때는 터미널이 끊길 수 있으므로 `nohup`이나 `screen/tmux` 환경에서 실행하는 것이 좋습니다.

```bash
#!/bin/bash

SOURCE="/A/20260318"
DEST="/B"

# 1. 대상 디렉토리 존재 확인
if [ ! -d "$SOURCE" ]; then
    echo "Source directory $SOURCE does not exist."
    exit 1
fi

# 2. 전송 실행
# --partial: 전송 중단 시 부분 전송된 파일을 보존하여 다음 실행 시 이어받기 지원
# --inplace: 대량 파일 작업 시 임시 파일을 만들지 않고 직접 덮어써서 디스크 공간 절약
echo "Starting file transfer with low I/O priority..."

ionice -c 3 nice -n 19 rsync -av \
    --partial \
    --inplace \
    --bwlimit=20000 \
    "$SOURCE" "$DEST"

echo "Transfer completed."
```

## 5. 작업 중단 및 재개 (Resume)

대량 전송 중 네트워크 문제나 수동 취소로 작업이 중단되어도 `rsync`는 중단된 지점부터 다시 시작할 수 있습니다.

### 5.1 이어받기 핵심 옵션
*   **`--partial`**: 전송이 중단되었을 때, 이미 전송된 데이터 조각을 삭제하지 않고 보관합니다.
*   **`--append-verify`**: (매우 유용) 파일의 크기를 비교하여 이미 전송된 부분 이후부터 데이터를 덧붙입니다. 대용량 파일 전송 중단 시 처음부터 다시 보내지 않아도 되어 매우 효율적입니다.

### 5.2 백그라운드 실행 (`nohup`)
장시간 작업 시 터미널 세션이 끊길 것을 대비해 `nohup`을 사용합니다.

```bash
nohup ionice -c 3 rsync -av --partial --append-verify --inplace --bwlimit=20000 /A/20260318 /B/ > transfer.log 2>&1 &
```

*   `> transfer.log 2>&1`: 모든 출력과 에러 메시지를 `transfer.log`에 기록합니다.
*   `&`: 백그라운드에서 실행합니다.
*   작업이 잘 돌아가는지 확인: `tail -f transfer.log`
